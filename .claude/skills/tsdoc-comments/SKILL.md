# tsdoc-comments

引き継ぐ人（人間・AI エージェント）のためのドキュメントコメントを書くための規約。
「コメントを書くこと」自体が目的ではない。**コードから読み取れない文脈を、次にこの
コードを触る人へ引き継ぐこと**が目的である。

## いつ・どこに書くか（対象と粒度）

export の有無（public / private）を問わず、以下のすべての宣言に TSDoc を付ける。

| 対象 | 最低限書くこと |
| --- | --- |
| クラス・関数・メソッド | 責務 + 契約（`@param` / `@returns` / `@throws`） |
| React コンポーネント | 責務・使いどころ。Props 型の各プロパティにも個別に TSDoc |
| カスタムフック | 責務・呼び出し制約（どの層で呼べるか、再レンダリングへの影響） |
| 型・インターフェース | その型が表すドメイン概念。非自明なプロパティには個別に TSDoc |
| 定数 | その値を選んだ理由と、変更したら何が起きるか |
| 設定値（`*.config.ts` など） | 何を設定するとどうなるか・なぜこの値か（非自明なもののみ） |

## 書くべき内容 — 4 分類

コメント本文は、次のいずれかに答えていなければ書く意味がない。

1. **契約** — 型で表せない事前条件・単位・値域・フォーマット・null や空のときの挙動・
   失敗時の振る舞い。
2. **意図** — なぜこの方針・アルゴリズム・値を選んだか。検討して却下した代替案と
   その理由。
3. **根拠** — どの仕様・issue・外部制約（API の制限、ブラウザの挙動など）に由来する
   判断か。出典（`specs/` 配下のパス、issue 番号、`@see` の URL）を示す。
4. **罠** — 変更すると壊れやすい点、非自明な依存、呼び出し順序の制約、
   パフォーマンス上の理由でこう書いてある箇所。

## 禁止事項

- **what の言い換え**を書かない。「スコアを計算する」「i をインクリメントする」のような
  コードの逐語的説明は違反。書くことがなければ契約だけを書く（後述の安全弁）。
- 実装と矛盾したコメントを残さない。コードを変更したら、影響を受けるコメントを
  **同じ変更の中で**修正または削除する。
- 変更履歴を書かない。「〜から変更」「以前は〜だった」は git が持つ情報。
- レビュー向けの正当化（「この修正は正しい」等）を書かない。コメントの読者は
  未来の保守者であり、今回のレビュアーではない。

## タグの使い分け

TSDoc 標準タグのみ使う（TypeScript では型情報が重複する `@type` 等の JSDoc 独自タグは
使わない）。

- `@param` — 型で表せない制約を書く。型の言い換えだけなら意味の説明を添える。
- `@returns` — 戻り値の意味・値域・特殊ケース（空配列、null の意味）。
- `@throws` — どの条件でどの例外が投げられるか。
- `@remarks` — 設計判断・却下した代替案・パフォーマンス上の理由など、要約に収まらない文脈。
- `@example` — 使い方が非自明な公開 API に付ける。
- `@see` — 根拠となる仕様（`specs/` 配下）・issue・外部ドキュメントへのリンク。
- `@defaultValue` — オプショナルなプロパティ・引数の既定値と、その値を選んだ理由。
- `@internal` — モジュール内専用であることを意図的に示す場合。

## 自明な宣言の扱い（安全弁）

「なぜ」が存在しない自明な宣言に、理由をでっち上げてはいけない。その場合は
**契約だけを 1 行で**書く。

```ts
/** おもちゃ一覧ページの 1 カードに表示するメタデータ。 */
export type ToyCardProps = { ... };
```

無理にひねり出した理由は、嘘のコメントと同じだけコードの信頼を損なう。

## 良い例・悪い例

### 関数 — 契約と根拠を書く

```ts
// ❌ 悪い: コードの言い換えでしかなく、読者に新しい情報がない
/** スコアを計算して返す */
export function calculateScore(answers: Answer[]): number { ... }

// ✅ 良い: 仕様判断の根拠と、型で表せない契約が書いてある
/**
 * 回答履歴からゲームスコアを算出する。
 *
 * 連続正解ボーナスは指数ではなく線形（+10/連鎖）で加算する。低年齢の
 * プレイヤーが 1 問の失敗でスコア差が開きすぎないようにするための仕様判断。
 *
 * @param answers - 出題順に並んだ回答履歴。空配列を許容し、その場合は 0 を返す。
 * @returns 0 以上の整数スコア。上限なし。
 * @see specs/003-keisan-game/spec.md の FR-012
 */
export function calculateScore(answers: Answer[]): number { ... }
```

### 定数 — 値を選んだ理由と変更時の影響を書く

```ts
// ❌ 悪い: 値の言い換え。なぜ 300 なのかが引き継がれない
/** 300 ミリ秒 */
const DEBOUNCE_MS = 300;

// ✅ 良い: 試行の結果と、変更するときの確認事項が書いてある
/**
 * 入力確定を待つデバウンス時間。
 *
 * 200ms ではかな漢字変換の途中で検索が走り候補がちらついた。350ms を超えると
 * 体感で「反応が遅い」ため 300ms とした。変更する場合は IME 入力での挙動を
 * 必ず手で確認すること。
 */
const DEBOUNCE_MS = 300;
```

### private 関数 — 罠（非自明な制約）を書く

```ts
// ❌ 悪い: private だからとコメントを省略している
function stepParticles(particles: Particle[], deltaMs: number): void { ... }

// ✅ 良い: 破壊的更新という異例な設計の理由が書いてある
/**
 * 1 フレーム分のパーティクル位置を更新する（モジュール内専用）。
 *
 * requestAnimationFrame から毎フレーム呼ばれるため、この関数内での配列や
 * クロージャの生成は禁止（GC スパイクでフレーム落ちする）。引数の particles を
 * 破壊的に更新しているのはこの制約のため。
 *
 * @param particles - 更新対象。呼び出し後は位置・速度が書き換わっている。
 * @param deltaMs - 前フレームからの経過時間。タブ非アクティブから復帰した直後は
 *   巨大な値が入るため、内部で 100ms にクランプする。
 */
function stepParticles(particles: Particle[], deltaMs: number): void { ... }
```

## 完了前のセルフレビュー

実装を完了と報告する前に、変更したファイルを見直して以下を確認する。

- [ ] すべての宣言（export / private を問わず）に TSDoc が付いているか。
- [ ] 各コメントは「契約・意図・根拠・罠」のいずれかに答えているか。
      コードの言い換えになっていないか。
- [ ] `@param` / `@returns` に、型で表せない制約（単位・値域・空のときの挙動）が
      書いてあるか。
- [ ] 仕様・issue 由来の判断に出典（`@see`・パス）があるか。
- [ ] 今回の変更で古くなったコメントが残っていないか。
